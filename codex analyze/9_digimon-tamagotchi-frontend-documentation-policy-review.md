# Digimon Tamagotchi Frontend 문서 정책 및 구조 설명 문서 리뷰

작성일: 2026-03-29

## 목적

이 문서는 현재 코드 구조와 프로젝트의 문서 정책을 비교해, 앞으로 `docs/REFACTORING_LOG.md`에 어떤 형식으로 변경 기록을 남기면 좋은지와 현재 구조 설명 문서에서 부족한 부분이 무엇인지 정리한다.

비교 기준은 두 가지다.

1. 프로젝트 정책이 요구하는 문서화 수준
2. 실제 코드 구조와 현재 문서 집합이 제공하는 정보의 정확도와 일관성

---

## 요약

현재 저장소의 문서 정책은 "주요 변경을 반드시 기록한다"는 점에서는 비교적 잘 지켜지고 있다. 특히 `docs/REFACTORING_LOG.md`는 날짜, 작업 유형, 목적/영향, 영향 파일을 남기는 습관이 이미 자리잡아 있다.

하지만 구조 설명 문서는 다음 문제가 크다.

- 기준 문서가 하나로 정리되어 있지 않다.
- 오래된 문서와 현재 코드가 충돌한다.
- 중복 문서가 존재한다.
- 저장소 모드, repository 패턴, lazy update 기준 파일 같은 핵심 아키텍처 사실이 문서마다 다르게 보인다.
- `REFACTORING_LOG`는 풍부하지만, "행동 변화"와 "분석 문서 추가"가 같은 수준으로 섞여 있어 검색성과 추적성이 떨어진다.

즉 지금 필요한 것은 문서를 더 많이 쓰는 것이 아니라, **기준 문서와 로그 형식을 더 명확히 정하는 것**이다.

## 현재 문서 정책과 실제 운영 방식 비교

## 정책 측면의 강점

프로젝트 가이드 기준으로 현재 문서화 정책은 다음을 요구한다.

- 모든 주요 변경사항을 `docs/REFACTORING_LOG.md`에 기록
- 날짜 포함
- 기능/버그 설명 포함
- 영향 파일 목록 포함
- 아키텍처 결정 근거 포함

이 요구는 방향 자체가 좋다. 특히 게임 규칙과 저장 구조가 복잡한 프로젝트에서는 "왜 이렇게 바뀌었는가"를 남기는 문화가 중요하다.

## 실제 운영에서 잘 지켜지는 부분

현재 `docs/REFACTORING_LOG.md`는 다음을 상당 부분 충족한다.

- 날짜 기준으로 누적된다.
- 기능 추가, 버그 수정, 문서 추가가 구분된다.
- 영향 파일이 함께 적힌다.
- 구현 배경과 목적을 같이 남긴다.

즉 "로그를 남기지 않는다"가 문제가 아니라, **로그의 스키마가 조금씩 흔들리고 있다는 것**이 문제다.

## 실제 운영에서 부족한 부분

### 1. 코드 변경과 분석 문서 추가가 같은 무게로 섞여 있다

지금 `REFACTORING_LOG`는 실제 제품 동작을 바꾼 변경과, 구조 분석 문서를 추가한 변경이 모두 같은 레벨로 쌓인다.

이 자체가 잘못은 아니지만, 나중에 "언제 실제 게임 규칙이 바뀌었는지"를 찾으려면 노이즈가 커진다.

### 2. 검증 정보가 없다

대부분의 항목에 다음 정보가 빠져 있다.

- 테스트 여부
- 수동 검증 범위
- Firebase/localStorage 영향 여부
- 저장 스키마 변경 여부

게임 프로젝트에서는 이 정보가 중요하다. 같은 "버그 수정"이라도 UI 텍스트 수정과 저장 구조 수정의 리스크는 다르다.

### 3. 문서 업데이트 범위가 명시적으로 연결되지 않는다

어떤 변경은 구조 문서나 메커니즘 문서도 함께 수정해야 하지만, 현재 로그는 주로 "영향 파일"만 적고 문서의 source of truth가 갱신되었는지는 명시하지 않는다.

즉 "코드는 바뀌었는데 기준 문서는 안 바뀐 상태"가 남기 쉽다.

## 현재 구조 설명 문서의 문제

## 1. 기준 구조 문서가 하나로 고정되어 있지 않다

현재 구조 설명 성격의 문서는 여러 개다.

- `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`
- `docs/STRUCTURE_ANALYSIS.md`
- `docs/CURRENT_CODE_STATUS_ANALYSIS.md`
- `README.md`
- `codex analyze/1_digimon-tamagotchi-frontend-structure-analysis.md`

문제는 이 문서들이 역할이 다르지 않고 일부는 사실상 겹친다는 점이다.

특히 `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`와 `docs/STRUCTURE_ANALYSIS.md`는 같은 내용을 거의 그대로 중복 보관하고 있다. 이런 구조에서는 "무엇이 최신인가"가 다시 모호해진다.

## 2. 루트 README가 현재 구조를 설명하지 못한다

루트 [README.md](/Users/hantaehee/Desktop/파일/개발/프로젝트/디지몬/d2/d2_tama_refact/README.md#L1)는 현재 앱 구조와 크게 어긋난다.

- `components/`가 "추후 추가"로 적혀 있다.
- 로그인, 다중 디지몬 관리, 진화, 배틀이 미래 계획처럼 적혀 있다.
- 실제로는 Hook, Firebase, 다수 페이지, 도감/배틀/아레나/모달 구조가 이미 존재한다.

즉 README는 신규 진입자가 보면 현재 프로젝트 상태를 오해하게 만든다.

## 3. 저장소 문서가 실제 런타임과 충돌한다

[src/repositories/README.md](/Users/hantaehee/Desktop/파일/개발/프로젝트/디지몬/d2/d2_tama_refact/digimon-tamagotchi-frontend/src/repositories/README.md#L1)는 repository 패턴과 `localStorage` 기본 모드를 전제로 설명한다.

하지만 [SlotRepository.js](/Users/hantaehee/Desktop/파일/개발/프로젝트/디지몬/d2/d2_tama_refact/digimon-tamagotchi-frontend/src/repositories/SlotRepository.js#L1)는

- localStorage 모드 제거
- Firebase만 사용
- 실제 코드에서는 이 repository를 직접 사용하지 않음

이라고 적고 있다.

즉 가장 위험한 종류의 문서 불일치가 이미 존재한다. "설계 문서"가 아니라 "잘못된 사용법 문서"가 되어 있다.

## 4. 게임 메커니즘 문서도 일부 현재 코드와 어긋난다

[docs/game_mechanics.md](/Users/hantaehee/Desktop/파일/개발/프로젝트/디지몬/d2/d2_tama_refact/docs/game_mechanics.md#L1)는 lazy update, 사망 조건, 액션 규칙을 설명하지만, 현재 코드와 완전히 동기화되어 있다고 보기 어렵다.

대표적으로:

- 사망 조건 서술이 현재 구현과 일부 다르다.
- 수명 사망 서술이 남아 있지만 실제 `useDeath`에는 제거되었다.
- 문서가 "공식 규칙"인지 "현재 구현"인지 구분이 없다.

이런 문서는 설명 자체보다도 **문서의 효력이 무엇인지**를 먼저 표시해야 한다.

## 5. 문서 인덱스와 상태 표시가 없다

`docs/`에는 분석서, 설계 메모, 버그 원인 분석, 구현 계획, 설명 문서가 매우 많다. 하지만 현재는 다음이 없다.

- `docs/README.md` 또는 `docs/INDEX.md`
- 문서 분류 기준
- 문서 상태 표시 (`draft`, `verified`, `stale`, `archived`)
- 기준 문서와 참고 문서의 구분

그래서 문서가 많을수록 탐색 비용이 늘어난다.

## 앞으로 `REFACTORING_LOG`에 남기면 좋은 형식

지금 로그의 장점은 살리고, 검색성과 추적성을 높이려면 형식을 조금만 더 표준화하면 된다.

## 권장 원칙

### 1. 한 항목은 "하나의 변경 단위"만 다룬다

- 실제 코드 동작 변경
- 문서 추가/갱신
- 데이터 구조 변경
- 운영 설정 변경

을 같은 항목에 섞지 않는 것이 좋다.

### 2. 행동 변화 여부를 명시한다

다음은 항상 보여야 한다.

- 사용자 동작 변화 있음/없음
- 저장 데이터 영향 있음/없음
- Firebase 경로 영향 있음/없음
- 문서만 변경인지 여부

### 3. 검증 방법을 한 줄이라도 남긴다

최소한 아래 중 하나는 있어야 한다.

- 테스트 실행
- 수동 확인
- 검증 미실시

이 정보가 없으면 나중에 로그를 읽는 사람이 리스크를 판단하기 어렵다.

### 4. 기준 문서 갱신 여부를 적는다

구조나 메커니즘이 바뀌면 단순히 "관련 파일 변경"만 적지 말고,

- 어떤 기준 문서를 같이 갱신했는지
- 혹은 아직 갱신하지 못했는지

를 적는 편이 좋다.

## 권장 템플릿

아래 정도면 충분하다.

```md
## [YYYY-MM-DD] 변경 제목

### 작업 유형
- 기능 추가 | 버그 수정 | 리팩터링 | 문서 | 데이터/스키마 | 운영

### 범위
- 사용자 동작 변화: 있음 / 없음
- 저장 데이터 영향: 있음 / 없음
- Firebase 영향: 있음 / 없음
- 문서만 변경: 예 / 아니오

### 배경
- 왜 이 변경이 필요했는지

### 변경 내용
- 핵심 구현 또는 수정 사항 2~5개

### 검증
- 테스트 실행: 예 / 아니오
- 수동 검증: 무엇을 확인했는지
- 미검증 리스크: 있으면 한 줄

### 영향받은 파일
- 경로 목록

### 문서 갱신
- 갱신한 기준 문서
- 후속 갱신이 필요한 문서
```

## 분석 문서 추가 전용 간단 템플릿

분석 문서는 더 짧게 가져가도 된다.

```md
## [YYYY-MM-DD] 분석 문서 제목

### 작업 유형
- 문서 추가

### 목적
- 어떤 질문에 답하기 위한 문서인지

### 산출물
- 생성한 문서 경로

### 비고
- 코드 변경 없음
```

즉 실제 제품 변경 로그와 분석 문서 로그는 형식을 분리하는 편이 좋다.

## 구조 설명 문서에서 앞으로 보강해야 할 내용

## 1. "현재 기준 문서"를 한 개로 지정해야 한다

추천 기준 문서:

- `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`

또는

- `docs/ARCHITECTURE_OVERVIEW.md` 같은 새 이름

이 문서는 반드시 아래를 담아야 한다.

- 엔트리 포인트
- 주요 페이지와 라우팅
- `Game.jsx`와 주요 Hook 책임
- 상태 경계
- 저장소 경계
- lazy update 기준 경로
- data/logic/repository 간 실제 관계

그리고 문서 첫 줄에 아래 메타 정보가 있으면 좋다.

- 마지막 검증 날짜
- 검증 기준 커밋 또는 브랜치
- 문서 상태 (`verified`)

## 2. 저장소 지원 범위를 명확히 적어야 한다

현재 가장 혼란스러운 주제 중 하나는 이것이다.

- Firebase와 localStorage를 둘 다 공식 지원하는가
- repository 패턴은 현재 실사용 경로인가
- `useGameData`가 저장의 기준 구현인가

이 질문에 대한 답이 문서마다 다르면 신규 작업자가 잘못된 가정을 갖게 된다.

따라서 구조 문서에는 다음 표가 필요하다.

| 주제 | 문서상 목표 | 현재 실제 구현 | 기준 파일 |
|------|------------|---------------|----------|
| 저장 모드 | dual storage | Firebase 중심 | `useGameData.js` |
| repository 패턴 | 추상화 계층 | 실사용 아님 | `SlotRepository.js` |
| lazy update | 핵심 패턴 | `data/stats.js`, `logic/stats/stats.js` 병존 | 둘 다 명시 |

## 3. "공식 규칙"과 "현재 구현"을 분리해서 적어야 한다

게임 메커니즘 문서는 아래 둘을 구분해야 한다.

- 원작/기획 기준 규칙
- 현재 코드 구현 규칙

이 둘이 섞이면 버그 분석인지 스펙 문서인지 구분이 안 된다.

문서 제목이나 섹션에서라도 다음을 분리하는 편이 좋다.

- `Spec`
- `Current Implementation`
- `Known Divergences`

## 4. 문서 분류 체계를 만들어야 한다

현재 `docs/`는 정보량은 많지만 구조가 평평하다. 최소한 아래 분류는 필요하다.

- `docs/architecture/`
- `docs/mechanics/`
- `docs/analysis/`
- `docs/runbooks/`
- `docs/archive/`

지금 바로 폴더를 바꾸지 않더라도, 최소한 `docs/INDEX.md`에서 분류와 추천 읽기 순서를 제공해야 한다.

## 5. 중복 문서를 정리하거나 관계를 명시해야 한다

예를 들어:

- `CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`
- `STRUCTURE_ANALYSIS.md`

처럼 성격이 같은 문서는 둘 다 유지할 이유가 약하다.

둘 다 보관해야 한다면 다음 중 하나는 적어야 한다.

- 어떤 문서가 최신본인지
- 어떤 문서가 superseded 되었는지
- 어떤 문서가 참고용 아카이브인지

이 순서가 좋은 이유는 "신규 진입자 오해를 줄이는 문서"부터 바로잡는 것이 가장 효과적이기 때문이다.

## 결론

현재 프로젝트는 문서가 부족한 저장소가 아니라, **문서가 많지만 기준과 상태 표시가 부족한 저장소**에 가깝다.

`REFACTORING_LOG`는 이미 좋은 습관이 자리잡아 있으므로 없애거나 크게 바꿀 필요는 없다. 대신 앞으로는 다음 네 가지만 강화하면 된다.

1. 코드 변경과 분석 문서 추가를 구분해서 기록하기
2. 행동 변화, 저장 영향, 검증 여부를 명시하기
3. 구조/메커니즘의 기준 문서를 하나로 정하기
4. 오래된 문서와 현재 코드가 충돌하는 지점을 빠르게 정리하기

즉 앞으로의 문서 정책 목표는 "더 많이 기록하기"가 아니라, **어떤 문서가 진실의 원천인지 더 분명하게 만들기**다.
