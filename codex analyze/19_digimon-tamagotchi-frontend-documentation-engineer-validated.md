# Digimon Tamagotchi Frontend Documentation Engineer Review

- 기준 관점: `documentation-engineer`
- 검토 범위:
  - `README.md`
  - `docs/REFACTORING_LOG.md`
  - `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`
  - `docs/game_mechanics.md`
  - `digimon-tamagotchi-frontend/src/repositories/README.md`
  - 현재 소스 구조와 기존 구조/아키텍처 분석 결과

## 1. 요약

현재 프로젝트의 문서 문제는 "문서가 부족하다"가 아니라 "문서의 역할이 구분되지 않고, 오래된 문서가 기준 문서처럼 남아 있다"는 점에 더 가깝다.

- 가장 큰 마찰은 실제 코드 구조와 문서가 서로 다른 계약을 설명한다는 점이다.
- `docs/REFACTORING_LOG.md`는 이미 좋은 습관을 갖고 있지만, 변경의 성격과 검증 상태를 더 분명히 기록할 필요가 있다.
- 구조 설명 문서는 존재하지만, 무엇이 "현재 기준 문서"이고 무엇이 "분석/참고 문서"인지 구분이 약하다.

## 2. 현재 코드 구조와 문서 정책 비교

### 강점

- `docs/REFACTORING_LOG.md`는 날짜, 작업 유형, 목적, 영향 파일을 일관되게 남기고 있다.
- `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`는 현재 코드 기준 구조를 비교적 충실하게 설명하고 있다.
- `codex analyze` 폴더에 분석 산출물이 누적되어 있어, 의사결정 맥락이 남는 편이다.

### 주요 불일치

#### 2.1 루트 README가 현재 프로젝트 상태를 대표하지 못함

루트 `README.md`는 현재 코드베이스를 소개하는 첫 문서여야 하지만, 실제로는 오래된 폴더 구조와 "향후 개발 예정" 항목을 그대로 포함하고 있다.

- 현재는 이미 존재하는 기능이 미구현처럼 보인다.
- 현재 `src/` 구조와 맞지 않는 설명이 남아 있다.
- 신규 기여자가 가장 먼저 읽는 문서가 잘못된 전제를 심어줄 가능성이 크다.

#### 2.2 저장소 구조 설명이 실제 런타임 계약과 다름

`digimon-tamagotchi-frontend/src/repositories/README.md`는 repository 패턴과 localStorage/Firestore 전환 구조를 기준 계약처럼 설명하지만, 실제 런타임은 해당 추상화를 중심으로 돌지 않는다.

- 문서는 `REACT_APP_STORAGE_TYPE` 기반 전환과 repository 사용을 전제로 쓴다.
- 실제 분석 결과에서는 페이지와 Hook가 Firebase를 직접 참조하는 경로가 더 강하다.
- 결과적으로 저장소 문서는 "현재 구현 설명"이 아니라 "과거 설계 혹은 미완 구조"에 가깝다.

#### 2.3 게임 규칙 문서가 사양과 구현을 분리하지 않음

`docs/game_mechanics.md`는 유용하지만, 현재 구현과 도메인 의도 중 무엇을 설명하는 문서인지 경계가 약하다.

- 일부 규칙 설명은 현재 코드와 다를 가능성이 있다.
- 구현 기준 문서인지, 디자인 문서인지, 포트폴리오 설명인지 명확하지 않다.
- 이런 문서는 시간이 지나면 가장 빨리 drift가 생긴다.

#### 2.4 구조 설명 문서는 있지만 canonical 문서 체계가 없음

`docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md` 같은 문서는 좋지만, "이 문서가 현재 구조의 기준 문서다"라는 운영 규칙이 없다.

- 비슷한 성격의 구조/상태/분석 문서가 여러 개 있다.
- 어떤 문서가 최신이고 어떤 문서가 참고용인지 구분이 약하다.
- `docs` 전체를 안내하는 인덱스 문서가 없다.

## 3. 문서 정책 측면의 핵심 문제

### 가장 큰 마찰

현재 문서 정책의 가장 큰 문제는 "모든 변경을 기록한다"는 규칙은 있으나, "어떤 문서가 기준 문서인지"에 대한 규칙은 약하다는 점이다.

즉, 기록은 잘 쌓이는데 기준은 약하다.

### 장기 유지보수 리스크

- 오래된 문서가 살아남아 신규 작업자가 잘못된 구조를 기준으로 판단할 수 있다.
- 분석 문서가 많아질수록 오히려 검색 비용이 커질 수 있다.
- `REFACTORING_LOG`가 계속 누적되어도, 기준 문서가 갱신되지 않으면 구조 drift를 막지 못한다.

## 4. 앞으로의 REFACTORING_LOG 권장 형식

현재 형식을 버릴 필요는 없다. 다만 "변경의 성격"과 "검증 상태"를 더 명확히 남기도록 표준 필드를 보강하는 편이 좋다.

### 4.1 코드/동작 변경용 권장 템플릿

```md
## [YYYY-MM-DD] 작업 제목

### 작업 유형
- 기능 추가 / 버그 수정 / 리팩터링 / 문서 갱신 / 분석 문서 추가

### 범위
- 사용자 동작 변화: 있음 / 없음
- 저장 데이터 영향: 있음 / 없음
- Firebase 영향: 있음 / 없음
- localStorage 영향: 있음 / 없음
- 문서만 변경: 예 / 아니오

### 배경
- 왜 이 변경이 필요했는지

### 변경 내용
- 핵심 변경 사항 2~5개

### 검증
- 자동 테스트: 실행함 / 미실행
- 수동 검증: 어떤 시나리오를 확인했는지
- 미검증 리스크: 남아 있는 부분

### 영향받은 파일
- 파일 목록

### 문서 갱신
- 갱신한 기준 문서
- 후속 갱신이 필요한 문서
```

### 4.2 분석 문서/문서 전용 변경용 축약 템플릿

```md
## [YYYY-MM-DD] 작업 제목

### 작업 유형
- 문서 추가 / 분석 문서 추가 / 정책 문서 갱신

### 목적
- 무엇을 정리했는지

### 산출물
- 생성/수정한 문서 목록

### 비고
- 코드 변경 없음
- 후속으로 갱신이 필요한 기준 문서
```

### 4.3 왜 이 형식이 좋은가

- `REFACTORING_LOG`만 봐도 "실제 동작이 바뀌었는지"를 바로 알 수 있다.
- 저장소/Firebase 영향 여부를 남기면 이 프로젝트의 민감한 변경을 빠르게 추적할 수 있다.
- 분석 문서와 코드 변경 로그를 같은 문서 안에서도 덜 혼동하게 된다.
- 후속 문서 갱신 누락을 줄일 수 있다.

## 5. 현재 구조 설명 문서에서 부족한 부분

### 5.1 문서 역할 분류가 부족함

필요한 구분은 최소 네 가지다.

- 기준 문서: 현재 구조/현재 규칙/현재 개발 방법
- 참고 문서: 상세 분석, 버그 조사, 회고
- 설계 문서: 의도한 방향과 목표 상태
- 보관 문서: 더 이상 기준이 아닌 과거 기록

지금은 이 분류가 디렉토리/파일명/문서 머리말 어디에도 명확히 드러나지 않는다.

### 5.2 문서 상태 메타데이터가 없음

구조 설명 성격의 문서에는 최소한 다음 메타데이터가 있으면 좋다.

- 상태: `current`, `reference`, `stale`, `archived`
- 마지막 코드 검증일
- 검증 범위
- 관련 기준 문서 링크

이 표식이 없으면 오래된 문서도 현재 문서처럼 보이게 된다.

### 5.3 시작점 문서가 약함

지금 프로젝트는 문서가 많지만 "처음 읽는 순서"가 없다.

권장되는 최소 시작점은 다음과 같다.

1. 루트 `README.md`
2. `docs/INDEX.md` 또는 `docs/README.md`
3. 현재 구조 기준 문서
4. 현재 게임 규칙 기준 문서
5. `docs/REFACTORING_LOG.md`

### 5.4 구조 문서와 실제 저장소 계약 연결이 부족함

이 프로젝트에서 가장 중요한 민감 경계는 다음이다.

- `Game.jsx`와 대형 Hook의 책임 경계
- Firebase와 localStorage의 실제 지원 범위
- repository 패턴의 실사용 여부
- lazy update의 canonical runtime source

하지만 현재 문서 체계에서는 이 네 가지가 여러 문서에 나뉘어 설명되거나, 서로 다르게 설명된다.

## 6. 가장 작은 안전한 개선안

가장 작은 비용으로 효과가 큰 순서는 아래와 같다.

1. 루트 `README.md`를 현재 구조 기준으로 정리
2. `docs/INDEX.md` 또는 `docs/README.md`를 추가해 문서 역할을 구분
3. `src/repositories/README.md`를 "현재 구현" 기준으로 수정하거나, 실험/레거시 문서임을 명시
4. `docs/game_mechanics.md`에 "현재 구현 기준"인지 "설계/의도 기준"인지 선언 추가
5. `REFACTORING_LOG.md`에 검증/영향 범위 필드 추가

## 7. 문서 운영 제안

### 기준 문서 후보

- 루트 `README.md`: 프로젝트 진입 설명
- `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`: 현재 구조 기준 설명
- `docs/game_mechanics.md`: 현재 규칙 또는 설계 규칙 설명
- `docs/REFACTORING_LOG.md`: 변경 이력

### 분석 문서 보관 원칙

`codex analyze`와 개별 분석 문서는 계속 유지해도 된다. 대신 기준 문서가 이 분석 문서들로 직접 대체되면 안 된다.

- 기준 문서는 짧고 명확해야 한다.
- 분석 문서는 상세 근거와 맥락을 담아야 한다.
- 기준 문서에서 분석 문서로 링크하는 구조가 좋다.

## 8. 이번 검토에서 확인한 사항

- 루트 `README.md`는 현재 코드 구조/기능 상태와 불일치하는 항목이 있다.
- `docs/REFACTORING_LOG.md`는 기록 습관은 좋지만 필드가 더 명확해질 여지가 있다.
- `docs/CURRENT_PROJECT_STRUCTURE_ANALYSIS.md`는 현재 구조 설명 문서로 가장 신뢰도가 높다.
- `digimon-tamagotchi-frontend/src/repositories/README.md`는 현재 런타임 계약과 차이가 크다.
- `docs/game_mechanics.md`는 구현 기준과 설계 기준을 더 명확히 나눌 필요가 있다.

## 9. 잔여 리스크

- 이 리뷰는 정적 문서/코드 비교 기준이며, 런타임 동작 재검증 문서는 아니다.
- 문서 정비 없이 분석 문서만 계속 추가되면 검색 비용과 혼선이 더 커질 수 있다.
- 특히 저장소 모드와 repository 패턴은 기준 문서가 빨리 정리되지 않으면 이후 모든 설명 문서에 drift를 퍼뜨릴 수 있다.

## 10. 결론

현재 프로젝트의 문서 정책은 "변경 기록은 열심히 남긴다"는 점에서는 강점이 있다. 하지만 장기 유지보수 관점에서 더 중요한 것은 "무엇이 현재 기준 문서인지"를 분명히 하는 일이다.

우선순위는 다음 순서가 가장 안전하다.

1. 루트 `README.md` 최신화
2. 문서 인덱스/상태 체계 추가
3. repository/storage 설명 최신화
4. 게임 규칙 문서의 구현/설계 경계 명시
5. `REFACTORING_LOG.md` 템플릿 보강
