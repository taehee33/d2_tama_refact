## 1. 현재 구조 요약

현재 런타임의 중심 흐름은 `App.jsx -> SelectScreen.jsx -> Game.jsx`다.

- `App.jsx`는 `AuthProvider`, 라우터, 페이지 진입 경로를 묶는다.
- `SelectScreen.jsx`는 슬롯 목록/생성과 진입을 담당한다.
- `Game.jsx`는 실제 게임의 오케스트레이터다.

`Game.jsx`는 다음 Hook들을 조립해 메인 플레이를 구성한다.

- `useGameState`: React 상태, 모달, 플래그, UI 상태의 중심
- `useGameData`: 슬롯 로드/저장, lazy update, Firestore 연동
- `useGameActions`: 먹이, 훈련, 배틀 등 주 액션
- `useGameLogic`: 수면/보조 계산/게임 진행 보조 로직
- `useGameHandlers`: UI 이벤트와 액션 연결
- `useGameAnimations`: 액션 연출과 상태 전이
- `useEvolution`: 진화 판정과 진화 실행
- `useDeath`: 사망/환생 관련 처리
- `useArenaLogic`: 아레나 설정 및 시작 흐름

도메인 규칙은 `src/logic`에도 존재하지만, 현재 런타임의 canonical source가 `logic`만으로 닫혀 있지는 않다.

- `src/data/v1/digimons.js`는 구조화된 디지몬 데이터의 기준 소스다.
- `src/data/v1/adapter.js`는 구 포맷과의 호환 계층이다.
- 시간 기반 스탯과 일부 게임 규칙은 `src/data/*`, `src/logic/*`, `src/hooks/*`, `Game.jsx`에 분산돼 있다.

저장소 경계는 문서와 실제 런타임 사이에 차이가 있다.

- 문서상으로는 Firebase/localStorage 이중 저장소와 repository 패턴이 강조된다.
- 실제 핵심 저장 경계는 `useGameData`와 여러 Hook 내부의 직접 Firestore 호출에 더 가깝다.
- `localStorage`는 현재 슬롯 영속성보다는 보조 UI 상태/개발 설정 쪽에 더 많이 남아 있다.

lazy update는 개념적으로는 프로젝트의 핵심 성능 패턴이며, 실제로도 세 지점에서 중요하게 쓰인다.

- 슬롯 로드 직후 누적 상태 적용
- 액션 직전 상태 보정
- 저장 직전 방어적 재보정

다만 실시간 1초 타이머 경로와 lazy update 경로가 완전히 같은 규칙 소스를 쓰지는 않는다.

## 2. 현재 개발 상태와 강점

이 프로젝트는 구조적 리스크가 분명하지만, 동시에 기능 성숙도가 높은 편이다.

### 2.1 기능 폭이 넓고 실제 플레이 루프가 살아 있다

이미 다음 기능이 하나의 앱 안에서 연결돼 있다.

- 육성
- 진화와 조그레스
- 아레나/배틀/퀘스트
- 수면, 냉장고, 케어미스, 사망
- 슬롯 관리와 도감

즉 초기 프로토타입이 아니라, 운영 가능한 게임 앱에 가까운 복합 상태 시스템이다.

### 2.2 분리하려는 방향성 자체는 맞다

- Hook 분해 시도가 충분히 진행되어 있다.
- `logic`, `hooks`, `components`, `data`, `pages`로 구조를 나누려는 의도가 분명하다.
- `data/v1`와 adapter 계층은 장기적으로 좋은 방향이다.

현재 문제는 "아예 구조가 없다"가 아니라 "좋은 방향으로 분해하던 중 경계가 완전히 정리되지는 않았다"에 가깝다.

### 2.3 lazy update 전략 자체는 맞다

시간 기반 게임에서 Firestore write를 실시간으로 밀어 넣지 않고, 액션/복귀 시점에 누적 적용하는 전략은 비용과 안정성 면에서 올바른 방향이다.

- Firestore write 비용을 줄일 수 있다.
- 오프라인 복귀와 재접속 시나리오에 대응할 수 있다.
- 장기적으로도 계속 유지해야 할 핵심 패턴이다.

### 2.4 구조화 데이터와 문서화 문화가 있다

- `src/data/v1` 구조는 데이터 주도형 확장을 가능하게 한다.
- `docs/REFACTORING_LOG.md`와 각종 분석 문서가 쌓여 있어, 의사결정 흔적이 남는 편이다.

즉 지금 상태는 "재작성"보다 "경계 정리와 안전망 강화"가 더 적절한 코드베이스다.

## 3. 핵심 리스크

아래는 우선순위가 높은 순서대로 정리한 현재 구조 리스크다.

### P1. 문서상의 아키텍처 계약과 실제 런타임 계약이 다르다

가장 먼저 정리해야 할 문제다.

- 문서는 dual storage, repository abstraction, localStorage 모드를 강조한다.
- 실제 핵심 흐름은 Firebase 중심이다.
- repository 패턴은 존재하지만, 현재 메인 런타임 경계라고 보기 어렵다.

이 불일치는 신규 작업자가 잘못된 전제를 갖게 만들고, 리팩터링 우선순위를 흐리게 만든다.

### P1. `Game.jsx`와 대형 Hook들이 page/controller/service/domain 책임을 동시에 가진다

현재 `Game.jsx`는 단순 페이지가 아니라 다음을 함께 가진다.

- Hook 조립
- 실시간 타이머
- 상태 파생
- 저장 흐름 연결
- 일부 직접 Firestore 처리
- UI 렌더 진입점

대형 Hook들도 책임이 완전히 분리돼 있지 않다.

- `useGameActions`: 액션 계산 + 저장 + 일부 영속성 부수효과
- `useEvolution`: 판정 + 실행 + UI/저장 보정
- `useGameHandlers`: UI 이벤트 + 도메인 트리거 + 저장 후 보정
- `useDeath`: 사망 관련 규칙 + 환생 흐름 + 일부 저장 연계

결과적으로 기능 하나를 바꿀 때 페이지와 여러 Hook를 동시에 이해해야 한다.

### P1. 게임 규칙의 canonical source가 하나가 아니다

correctness와 regression risk 관점에서 가장 위험한 문제다.

- 시간 기반 스탯 계산이 `src/data/stats.js`와 `src/logic/stats/stats.js`에 나뉜다.
- 훈련/먹이/사망/케어미스 관련 규칙도 Hook와 page에 일부 중복된다.
- 문서에 적힌 규칙과 현재 구현도 완전히 일치한다고 보기 어렵다.

이 상태에서는 "규칙 수정"이 아니라 "규칙 구현 후보 중 하나 수정"이 되기 쉽다.

### P1. 실시간 경로와 lazy update 경로가 같은 플레이 결과를 보장하지 못할 수 있다

게임 플레이 품질과 직결되는 문제다.

특히 다음 항목에서 drift 가능성이 크다.

- 배고픔/힘 감소 누적
- 0 상태 지속 시간 추적
- 케어미스 재구성
- 수면/수면 방해
- 냉장고 시간 제외
- 배변/부상/사망
- 진화 대기 시간

이 부분이 경로마다 다르면 "오래 켜 두고 플레이한 경우"와 "나갔다 다시 들어온 경우"의 결과가 달라질 수 있다.

### P1. 자동 테스트가 거의 없어 고위험 규칙 변경을 방어하지 못한다

현재 구조에서는 작은 리팩터링도 곧 회귀 위험이다.

가장 취약한 영역은 다음이다.

- `applyLazyUpdate`
- 진화 조건
- 배틀 계산과 명중률
- 사망 조건
- 저장 시점
- 모달 전이

이 영역은 규칙이 복잡한데도 자동 검증이 거의 없다.

### P2. 저장 시점과 Firestore write 경계가 분산돼 있다

저장은 `useGameData` 중심으로 보이지만 실제로는 여러 곳에 분산돼 있다.

- 일부 Hook가 Firestore를 직접 갱신한다.
- 슬롯 본문, 로그, 도감, 배경 설정 등이 서로 다른 경로로 저장된다.
- 실패 시 partial update와 write 비용 증가 가능성이 있다.

이 구조는 성능 문제이기도 하고, 구조 문제이기도 하다.

### P2. 성능 리스크는 루트 리렌더, 파생 계산, Firestore I/O에 집중돼 있다

지금 당장 치명적이라기보다, 기능이 늘수록 누적될 가능성이 큰 영역이다.

- `Game.jsx`의 1초 타이머들
- 넓은 상태 덩어리와 광범위한 props 전달
- 실시간 UI 업데이트가 큰 트리 리렌더로 전파되는 구조
- 액션당 `getDoc + updateDoc + addDoc + snapshot 재수신` 패턴

lazy update 전략이 있음에도, 주변 구조 때문에 비용 이점이 일부 상쇄될 수 있다.

### P2. 모달 상태 관리가 중앙 객체 패턴이지만, 전이 규칙은 명시적이지 않다

`modals` 객체로 모은 것은 장점이지만, 어떤 모달이 어떤 상태와 동시에 열리거나 닫혀야 하는지 명시적 상태 전이 모델은 약하다.

- 액션 연출/모달 종료/저장 타이밍이 서로 얽힐 수 있다.
- UI 회귀가 나도 자동 테스트로 잡기 어렵다.

## 4. 개선이 필요한 부분

현재 프로젝트에서 가장 필요한 개선은 "더 많이 쪼개기"보다 "경계를 믿을 수 있게 만들기"다.

### 4.1 런타임 계약을 먼저 명확히 해야 한다

우선 다음 질문부터 정리해야 한다.

- 실제로 localStorage 오프라인 모드를 계속 지원할 것인가
- repository 패턴을 실제 런타임 경계로 복구할 것인가
- 아니면 Firebase 중심 구조를 공식 계약으로 선언할 것인가

이 결정이 정리되지 않으면 이후 리팩터링도 반쯤은 가짜 경계를 기준으로 진행된다.

### 4.2 `Game.jsx`의 조립 책임과 실행 책임을 나눠야 한다

현재 필요한 것은 거대한 파일을 무작정 쪼개는 것이 아니라, 다음 책임을 분리하는 일이다.

- page/presenter 책임
- runtime controller 책임
- timer/subscription 책임
- persistence adapter 책임
- action composition 책임

### 4.3 규칙 엔진의 단일 기준 소스를 만들어야 한다

장기적으로는 다음이 필요하다.

- 시간 기반 스탯 계산의 단일화
- 사망 조건의 단일화
- 훈련/먹이/케어미스 규칙의 단일화
- 진화/배틀 계산의 테스트 가능한 순수 함수화 유지

### 4.4 저장 경계와 저장 정책을 일원화해야 한다

현재는 "누가 저장 권한을 가지는가"가 흐리다.

필요한 방향은 다음이다.

- 저장 진입점 최소화
- Firestore 직접 접근 위치 축소
- 슬롯 데이터와 부가 데이터의 저장 정책 문서화
- save 전에 어떤 보정을 하는지 일관화

### 4.5 성능 최적화는 구조 정리 뒤에 정확히 들어가야 한다

지금 필요한 것은 성급한 memoization이 아니라 다음이다.

- 타이머 영향 범위 축소
- 큰 props 조립 분리
- 렌더와 파생 계산 경계 정리
- Firestore I/O 패턴 단순화

### 4.6 테스트와 문서가 리팩터링의 선행조건이어야 한다

이 코드베이스에서는 테스트와 문서가 부가 작업이 아니라 리팩터링 전제조건이다.

- characterization 테스트 없이 규칙 정리를 시작하면 회귀 가능성이 높다.
- 구조 문서 최신화 없이 리팩터링을 진행하면 잘못된 계약이 계속 남는다.

## 5. 우선순위별 리팩터링/테스트/문서화 제안

### 우선순위 0. 계약 정리와 회귀 기준선 확보

목표는 "무엇이 현재 기준인지"를 먼저 고정하는 것이다.

리팩터링:
- 아직 큰 구조 변경은 하지 않는다.
- `Game.jsx`와 주요 Hook의 현재 책임을 문서 기준으로 고정한다.

테스트:
- `applyLazyUpdate` characterization 테스트부터 추가
- 진화 조건 테스트 추가
- 배틀 계산/명중률 테스트 추가
- 사망 조건은 현재 구현 기준 characterization 테스트로 고정

문서화:
- 루트 `README.md` 최신화
- 현재 구조 기준 문서와 참고 문서 구분
- Firebase/localStorage/repository 실제 계약 명시

### 우선순위 1. `Game.jsx`를 안전하게 가볍게 만들기

목표는 페이지를 찢는 것이 아니라, 조립과 렌더 책임을 분리하는 것이다.

리팩터링:
- `Game.jsx`에서 props 조립 블록 추출
- presenter 성격의 렌더 블록 분리
- timer/subscription effect 분리

테스트:
- 주요 화면 전환 smoke test
- 모달 열림/닫힘과 액션 연계 테스트 최소 세트 추가

문서화:
- `Game.jsx` 책임 경계 문서 추가 또는 기존 구조 문서 갱신

### 우선순위 2. 저장 경계 단순화

목표는 저장 정책을 한 곳으로 수렴시키는 것이다.

리팩터링:
- direct Firestore 호출 지점 목록화
- `useGameData` 또는 별도 persistence adapter로 저장 경로 수렴
- 슬롯 본문 저장과 부가 로그 저장의 정책 분리

테스트:
- 액션 후 저장 payload 검증 테스트
- lazy update 후 save 시점 테스트
- write 누락/중복 방지 회귀 테스트

문서화:
- 저장 시점, 저장 필드, 부가 데이터 정책 문서화

### 우선순위 3. 규칙 엔진 단일화

목표는 플레이 결과를 경로와 무관하게 만드는 것이다.

리팩터링:
- 시간 기반 스탯 엔진 단일화
- 사망 조건 판정 단일화
- 훈련/먹이/케어미스 계산 경로 정리

테스트:
- online/offline 동일 결과 보장 테스트
- 수면/냉장고/0 상태 지속 시간 엣지 케이스 테스트
- 진화/사망 경계 케이스 확장

문서화:
- 현재 게임 규칙 문서를 "구현 기준"과 "설계 기준"으로 구분

### 우선순위 4. 성능 최적화와 구조 마감

목표는 최적화 포인트를 구조 경계 위에서 처리하는 것이다.

리팩터링:
- 타이머 영향 범위 축소
- 큰 상태/props 트리 축소
- Canvas/애니메이션 렌더 경계 점검
- snapshot 구독 범위 재검토

테스트:
- idle 상태 리렌더 관찰용 테스트/계측
- Firestore write 횟수 회귀 모니터링

문서화:
- lazy update 성능 원칙
- Firestore write 금지 규칙과 예외 정책

## 결론

현재 프로젝트는 기능적으로는 충분히 성숙하고, 분해 방향도 나쁘지 않다. 다만 지금 가장 큰 문제는 "분해된 것처럼 보이지만, 실제 경계는 아직 흐리다"는 점이다.

따라서 가장 안전한 순서는 다음이다.

1. 실제 계약과 기준 문서 정리
2. 시간 기반 규칙/진화/배틀 테스트 확보
3. `Game.jsx` 조립 책임 분리
4. 저장 경계 일원화
5. 규칙 엔진 단일화
6. 성능 최적화와 문서 체계 마감

이 순서를 따르면 behavior preserving 원칙을 지키면서도, 가장 큰 회귀 리스크부터 줄여갈 수 있다.
